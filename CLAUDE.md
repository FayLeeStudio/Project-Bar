## 这是什么（项目目标）

把"进度条"变成"赛道 + 赛车"。记录某件事的完成比例，用赛车在俯视赛道上
的位置来可视化它——跑完一圈 = 任务 100% 完成。比纯进度条更直观、更有趣。

最终目标支持多人："几个人约定好比什么、怎么算数据，然后各自的车同台竞速。"

---

## 核心架构原则（不要违反）

**赛道 = 一条 SVG path；车 = 这条 path 上 progress% 处的点。**

- 车的位置永远通过 `path.getPointAtLength((pct/100) * path.getTotalLength())` 求得。
- 想换赛道形状（椭圆 → 笔 → 一笔画 → 任意主题造型），**只改 `#trackPath` 的 `d` 属性**，
  车的逻辑、进度逻辑一律不动。
- 这是简单版和未来高级版共用的同一套骨架。任何改动都不能破坏这个抽象。

**全部应用状态集中在一个 `state` 对象里。**
- 阶段 1 是单个本地 state。阶段 2 会变成"一个 state 数组（每个玩家一个）"，
  渲染逻辑要能直接 map 这个数组，不要把单人逻辑写死到无法扩展。

---

## 技术栈（定死，不要随意更换）

- **纯 HTML + CSS + 原生 JavaScript，无框架、无构建步骤。**
- 单文件 `index.html`，双击就能在浏览器打开运行。
- 赛道用 **SVG**（因为它原生支持任意路径 + getPointAtLength）。
- **阶段 1 不引入任何外部依赖。** 不上 npm、不上 React、不上打包器。
- 理由：项目要轻量、好理解、好维护。保持"文件少、打开即跑"。

> 如果某个功能让你觉得"必须上框架/依赖了"，先停下来在对话里和 Fay 确认，
> 不要自作主张引入。

---

## 开发阶段（路线图）

- **阶段 1（当前）✅** 本地单人版。一条环形赛道 + 一辆小车（圆点带描边）+
  进度滑块。8-bit 像素风。进度只存在页面内存里，刷新清零。
- **阶段 2** 云端轻量多人。借现成云服务（如 Firebase 类）同步进度，
  多辆车同台竞速。**尽量轻量**，不自建后端服务器。
- **未来扩展点（先不做，但架构上留口子）：**
  - 赛道皮肤系统（不同视觉风格）
  - 自定义赛道形状（把赛道做成一支笔 / 与主题相关的一笔画造型）
  - 这些都应该只是"换 path 数据 / 换样式变量"，不该触动核心逻辑。

---

## 代码约定

- 注释和面向用户的文案用中文；代码标识符（变量/函数名）用英文。
- 颜色全部走 CSS 变量（`:root` 里定义），不要在代码里散落硬编码色值——
  这是皮肤系统的基础。
- 像素风：禁用字体平滑、用硬边框 + 方形阴影、`image-rendering: pixelated`。
- 保持可访问性底线：键盘可聚焦、`:focus-visible` 有可见描边、
  尊重 `prefers-reduced-motion`。
- 小步提交（git commit），每个功能一个清晰的提交，方便回退。
